# 2026-07-13 问题追踪

## 背景

本轮问题集中在文件管理页、索引过程可观测性、LightRAG 文件标识冲突，以及前端检索结果状态持久化。

## P1. 全部文件页面在 indexing 状态时无法长期留存

- 现象：用户选择“全部文件”后，文件正在 indexing 时页面会回到某个具体文件夹，无法长期停留在全部文件视图。
- 初步判断：前端用 `null` 表示“全部文件”，但刷新文件夹列表时会把空值自动替换成第一个文件夹。
- 修复方向：使用稳定 sentinel 表示“全部文件”，刷新时保留该状态；上传时再将 sentinel 转为默认未归档。
- 验收：处于 indexing/polling 时，用户选择“全部文件”不会被自动切走。

## P2. 文件 indexing 结束后出现一段时间长卡顿

- 现象：文件索引结束后，前端像暂时断开数据库一样卡顿。
- 初步判断：调度任务在 finally 中执行 LightRAG `finalize_storages()` 后才提交文件最终状态和释放锁；finalize 阶段可能较慢，导致数据库状态迟迟不更新。
- 修复方向：文件处理完成后尽早提交 `completed/failed` 状态；将 LightRAG finalize 放在状态提交之后，避免页面等待状态落库。
- 验收：文件完成后能较快从 `processing/indexing` 变为 `completed/failed`，不会长时间空白或无响应。

## P3. 不同文件夹下同名文件无法 indexing

- 现象：不同文件夹下上传同名文件后，索引失败。
- 初步判断：LightRAG 写入时 `file_paths=filename`，同名文件可能触发 LightRAG 内部文件路径/文档去重或冲突。
- 修复方向：传给 LightRAG 的 `file_paths` 使用包含 `file_id` 的稳定唯一显示名，例如 `{file_id}_{filename}`；业务响应仍显示原始 filename。
- 验收：不同文件夹下同名文件可分别索引成功。

## P4. indexing 状态时后端无日志

- 现象：后端只看到 LightRAG 自身零散日志，应用层没有明确“当前处理到哪里”的日志。
- 初步判断：SchedulerService 和 LightRAG 适配层未输出应用层 structured/progress 日志。
- 修复方向：在文件开始处理、解析完成、LightRAG 写入开始/完成、文件完成/失败等关键节点增加后端日志。
- 验收：后端控制台能看到 file_id、filename、阶段、chunk 数和失败原因。

## P5. 用户检索后刷新页面，检索结果消失

- 现象：刷新浏览器后，检索输入、结果、增强回答、图谱和 trace 全部丢失。
- 初步判断：前端检索状态只保存在 Vue 内存中。
- 修复方向：将最近一次检索状态写入 `localStorage`，页面加载时恢复；恢复失败则安全忽略。
- 验收：用户检索或 query 后刷新页面，仍能看到最近一次结果。

## 本轮落地记录

- P1：前端使用 `__all__` 作为“全部文件”的稳定状态，不再用 `null`；轮询刷新文件夹时保留“全部文件”选择。
- P2：调度任务在 finally 中先提交 `SchedulerLog` 汇总状态，再执行可能较慢的 LightRAG finalize；文件自身状态仍在 `_process_file()` 完成时立即提交。
- P3：写入 LightRAG 时 `file_paths` 从原始 `filename` 改为 `{file_id}_{filename}`，避免不同文件夹同名文件在 LightRAG 内部路径上冲突。
- P4：新增应用层日志，覆盖文件开始处理、解析完成、LightRAG 写入开始/完成、doc status、失败原因、任务汇总和 finalize 阶段。
- P5：前端将最近一次工作台检索/query 状态写入 `localStorage`，刷新页面后恢复 query 输入、增强回答、检索片段、图谱和 trace。
